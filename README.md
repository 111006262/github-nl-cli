# GitHub Natural Language Repository Search CLI

## Part 1: Build a Tool, Break It, and Harden It

## Project Overview

This project is a command-line interface (CLI) tool that converts a natural language query into a structured GitHub repository search request, executes the request using the GitHub API, and returns real repository results.

The goal of the project is to explore how natural language can be converted into structured API parameters. The tool also includes a "break-it" phase, where ambiguous, misspelled, conflicting, and multilingual inputs are tested. After identifying these failure cases, the parser is hardened with validation rules, typo fixes, basic Chinese keyword translation, and warnings for impossible or vague queries.

Example user input:

```bash
python cli.py "Find the top 5 Python repositories about machine learning"
```

Example structured GitHub API request generated by the tool:

```json
{
  "q": "machine learning language:Python",
  "sort": "stars",
  "order": "desc",
  "per_page": "5"
}
```

The tool then sends this request to GitHub and prints the top matching repositories in a JSON format.

---

## Domain Choice

The chosen domain is the GitHub public repository dataset.

I chose GitHub because its repository search has useful structured fields that can be extracted from natural language, such as:

- programming language
- topic/search terms
- number of stars
- creation date
- update date
- license
- number of returned results
- sorting method

This then makes GitHub a good target for a natural-language-to-structured-query tool.

## File-by-File Explanation

### `cli.py`

`cli.py` is the main command-line interface file.

It is responsible for:

1. Reading the user’s natural language query from the terminal.
2. Sending that query to `parser.py`.
3. Printing the structured GitHub API request.
4. Checking for warnings.
5. Sending the structured query to `github_client.py`.
6. Printing the final GitHub repository results.

Example command:

```bash
python cli.py "Find the top 5 Python repositories about machine learning"
```

The CLI first prints the original natural language input:

```text
Natural language input:
  Find the top 5 Python repositories about machine learning
```

Then it prints the structured GitHub API request:

```json
{
  "q": "machine learning language:Python",
  "sort": "stars",
  "order": "desc",
  "per_page": "5"
}
```

If the query encounters problems such as being too vague or having conflicting constraints, the CLI stops before calling the GitHub API.

For example:

```bash
python cli.py "Show me popular repos"
```

This query is too vague because it does not include a topic, language, or useful constraint. The tool therefore prints a warning and stops.

---

### `parser.py`

`parser.py` is the core natural language parser.

Its job is to convert the user’s natural language query into a structured query object. The parser is rule-based, which means it does not use an LLM. Instead, it uses regular expressions, dictionaries, and manually written extraction rules.

The main data structure is:

```python
@dataclass
class ParsedGitHubQuery:
    original_query: str
    search_terms: str
    language: Optional[str] = None
    min_stars: Optional[int] = None
    max_stars: Optional[int] = None
    created_after: Optional[str] = None
    created_before: Optional[str] = None
    pushed_after: Optional[str] = None
    license_name: Optional[str] = None
    exclude_terms: List[str] = field(default_factory=list)
    sort: str = "stars"
    order: str = "desc"
    per_page: int = 10
    warnings: List[str] = field(default_factory=list)
```

This object stores all extracted categories from the natural language query.

For example, the query:

```text
Find the top 5 Python repositories about machine learning with more than 1000 stars
```

is parsed into:

```text
search_terms = "machine learning"
language = "Python"
min_stars = 1000
sort = "stars"
order = "desc"
per_page = 5
```

Then the parser converts this into GitHub search syntax:

```text
machine learning language:Python stars:>1000
```

With this, the parser separates the query into categories using different helper functions.

#### `normalize_text()`

This function prepares the query before extraction.

It performs:

- basic whitespace cleanup
- simple typo correction
- simple Chinese keyword translation

For example:

```text
Find pyton repos about machien lerning
```

becomes:

```text
Find python repositories about machine learning
```

The parser includes dictionaries such as `TYPO_FIXES` and `CHINESE_TRANSLATIONS` to support some failure cases.

#### `extract_limit()`

This function detects how many results the user wants.

Examples:

```text
top 5
first 10
show 3
find 8
```

If a number is found, it becomes the `per_page` value. If no number is found, the default is 10.

#### `extract_language()`

This function detects programming languages.

It checks the query against the `LANGUAGES` dictionary. For example:

```text
Python
JavaScript
TypeScript
Rust
Go
C++
```

If the user writes:

```text
Find Rust repositories about blockchain
```

the parser extracts:

```text
language = "Rust"
```

Note that special handling is included for `Go` because `go` can also be an ordinary English verb.

#### `extract_stars()`

This function detects star constraints.

Examples:

```text
more than 1000 stars
over 500 stars
at least 200 stars
less than 100 stars
under 50 stars
```

For example:

```text
Find Python repositories with more than 1000 stars
```

becomes:

```text
stars:>1000
```

#### `extract_dates()`

This function detects date-related constraints.

Examples:

```text
created after 2021
created before 2020
updated this year
last year
```

For example:

```text
Find Rust blockchain projects created after 2021
```

becomes:

```text
created:>2021-12-31
```

#### `extract_license()`

This function detects license constraints.

Examples:

```text
MIT license
Apache license
GPL license
BSD license
```

For example:

```text
Find repositories about deep learning with MIT license
```

becomes:

```text
license:mit
```

#### `extract_exclusions()`

This function handles negative constraints.

Examples:

```text
not machine learning
without tensorflow
```

For example:

```text
Find AI repositories but not machine learning
```

becomes:

```text
ai -machine-learning
```

#### `infer_sort()`

This function decides how the GitHub results should be sorted.

Examples:

```text
newest
recently updated
most forks
forked
```

If no special sorting phrase is found, the default sorting method is by stars in descending order.

#### `clean_search_terms()`

After all structured fields are extracted, this function removes command words and filter words from the query. The remaining words become the main search topic.

For example:

```text
Find the top 5 Python repositories about machine learning with more than 1000 stars
```

The parser removes:

```text
find
top 5
python
repositories
about
with more than 1000 stars
```

The remaining topic is:

```text
machine learning
```

#### `validate_query()`

This function is the hardening layer for failure test cases.

It checks for:

- vague queries
- conflicting star constraints
- conflicting date constraints
- invalid result limits
- GitHub per-page limit issues

For example:

```text
Find repositories with more than 5000 stars and less than 100 stars about AI
```

This is impossible because a repository cannot have both more than 5000 stars and fewer than 100 stars. The parser detects this and adds a warning instead of blindly sending a bad request.

### `github_client.py`

`github_client.py` is responsible for communicating with the GitHub API.

It receives structured API parameters from `parser.py` and sends them to:

```text
https://api.github.com/search/repositories
```

The main function is:

```python
search_repositories(params)
```

This function:

1. Creates request headers.
2. Optionally uses a GitHub token from the environment.
3. Sends a GET request to GitHub.
4. Raises an error if GitHub returns a non-200 response.
5. Returns the JSON response.

The file also includes:

```python
simplify_results(api_response)
```

GitHub returns a large JSON object, so this function keeps only the fields needed for display:

- repository full name
- description
- stars
- forks
- programming language
- repository URL
- last updated time

This makes the CLI output easier to read.

Example output:

## Break-It Phase

After the baseline version worked, I tested the parser with difficult inputs as shown below.

| Test Input | Type of Difficulty | Expected Challenge |
|---|---|---|
| `Find pyton repos about machien lerning` | Typos | The parser must correct misspelled words |
| `找出關於機器學習的 Python 專案` | Non-English input | The parser must handle Chinese keywords |
| `Show me popular repos` | Ambiguous input | The parser should not guess randomly |
| `more than 5000 stars and less than 100 stars` | Conflicting constraints | The parser should detect impossible logic |
| `Find AI repositories but not machine learning` | Negative constraint | The parser should preserve exclusion |
| `Find top 200 Python repositories about data science` | API limit issue | The parser should cap the result count |
| `Find Go repositories about web servers` | Ambiguous word | The parser must distinguish Go as a language |

## Harden-and-Fix Phase

The parser was improved to handle the most important failure cases, mostly from typos to simple chinese to english translations, as well as ambiguous inputs.

### 1. Typo Handling

The parser includes a small typo correction dictionary.

Example:

```text
pyton → python
machien → machine
lerning → learning
```

Input:

```text
Find pyton repos about machien lerning
```

Corrected interpretation:

```text
machine learning language:Python
```

This makes the tool more robust against common spelling mistakes.

---

### 2. Basic Chinese Keyword Support

The parser includes a small Chinese-to-English keyword dictionary.

Examples:

```text
機器學習 → machine learning
人工智慧 → artificial intelligence
深度學習 → deep learning
電腦視覺 → computer vision
專案 → repository
```

Input:

```text
找出關於機器學習的 Python 專案
```

Parsed meaning:

```text
machine learning language:Python
```

This does not provide full multilingual understanding, but it supports common Chinese examples relevant to the GitHub repository search domain.

---

### 3. Ambiguity Detection

The tool detects vague queries.

Example:

```text
Show me popular repos
```

This query does not include a topic, language, star condition, date condition, or any useful search constraint. Instead of sending a weak search request to GitHub, the parser adds a warning:

```text
The query is too vague. Please include a topic, language, or constraint.
```

The CLI then stops and asks for clarification.

---

### 4. Conflict Detection

The parser detects impossible star constraints.

Example:

```text
Find repositories with more than 5000 stars and less than 100 stars about AI
```

This creates:

```text
min_stars = 5000
max_stars = 100
```

Since the minimum is greater than the maximum, the parser adds a warning:

```text
Conflicting star constraints: minimum stars is greater than or equal to maximum stars.
```

The CLI stops before calling GitHub.

---

### 5. Result Limit Protection

GitHub search requests should not ask for unlimited results in a single call. The parser protects against this by capping the result count.

Example:

```text
Find top 200 Python repositories about data science
```

The parser caps the result limit to 100.

---

### 6. Sorting Improvements

The parser supports different ranking intentions.

Examples:

```text
newest TypeScript chatbot repositories
```

Sorts by:

```text
created desc
```

```text
recently updated computer vision repositories
```

Sorts by:

```text
updated desc
```

```text
most forked Python repositories
```

Sorts by:

```text
forks desc
```

If the query does not specify a special sorting method, the default is:

```text
stars desc
```

This means the parser treats popularity as most-starred repositories.

## Remaining Failure Cases and Why They Are Fundamentally Hard

Even after hardening the parser, some failure cases remain difficult to solve completely, especially without an LLM.

### 1. Subjective Queries

Example:

```text
Find the best repositories for beginners learning AI.
```

This is difficult because words like `best` and `beginner-friendly` are subjective. The GitHub API can sort search queries by measurable fields like stars, forks, creation date, or update date, but it cannot directly determine whether a repository is truly beginner-friendly.

A repository with many stars is not always the best for beginners. It may be popular but poorly documented, outdated, or too advanced. Solving this properly would require deeper analysis of README files, tutorials, documentation quality, issue activity, and possibly user feedback.

Therefore, this is fundamentally hard because the user’s intent depends on qualitative judgment, while the GitHub API mostly supports quantitative filters.

---

### 2. Complex Semantic Similarity

Example:

```text
Find repositories similar to TensorFlow but easier to use.
```

This is hard because GitHub search does not directly support semantic similarity. The parser can search for the word `TensorFlow`, but it cannot truly understand what makes another repository similar to TensorFlow.

The phrase `easier to use` is also difficult because usability is not a simple structured field in the GitHub API. Solving this would require repository embeddings, README analysis, documentation analysis, or comparison against known machine learning frameworks.

This is fundamentally hard because the required information is not directly available as a simple API filter.

---

### 3. Full Multilingual Understanding

Example:

```text
Pythonで機械学習の人気リポジトリを探して
```

The current parser only supports a small dictionary of Chinese keyword translations. It does not support full Japanese, Korean, Indonesian, French, or mixed-language natural language queries.

Full multilingual support is difficult because users may phrase the same request in many different ways. A dictionary-based approach cannot cover all grammar patterns, synonyms, and languages.

Solving this properly would require a translation model, multilingual LLM, or a much larger multilingual parser.

---

### 4. Ambiguous Words

Example:

```text
Find Go projects about web servers.
```

In this case, `Go` probably means the Go programming language. However, in other sentences, `go` could simply be an English verb.

Example:

```text
I want to go find repositories about AI.
```

A simple rule-based parser may incorrectly interpret `go` as the programming language. The current parser includes special handling for Go, but ambiguous words are still generally difficult.

This is fundamentally hard because correct interpretation depends on sentence context, not just keyword matching.

---

### 5. Multi-Intent Queries

Example:

```text
Find Python machine learning repositories that are actively maintained, beginner-friendly, and not too large.
```

This query contains multiple abstract requirements:

- actively maintained
- beginner-friendly
- not too large

The parser can approximate `actively maintained` using `recently updated`, but it cannot fully judge maintenance quality. It also cannot determine beginner-friendliness or project size without deeper analysis of repository contents.

This is hard because the query asks for judgments that are not directly represented in the GitHub search API.

---

### 6. Hidden User Assumptions

Example:

```text
Find good repositories for my final project.
```

This is difficult because the tool does not know what the user’s final project is about. The query is meaningful to the user, but not specific enough for the parser.

There is usually a follow-up question to this, such as:

```text
What is your final project topic?
```

The current CLI only warns when the query is too vague. It does not have a full conversation system for asking follow-up questions.

This is fundamentally hard because the missing information exists outside the query itself.

---

## Technical Design Trade-Offs

This project uses a rule-based parser instead of an LLM for Part 1.

The advantage of this approach is that it is:

- explainable
- easy to debug
- deterministic
- inexpensive
- simple to run locally

The disadvantage is that it is less flexible than an LLM. It can only handle the patterns that were manually implemented.

For example, the parser can understand:

```text
more than 1000 stars
```

but may fail on unusual phrasings such as:

```text
repos that have gained at least a thousand stars
```

A future version could use an LLM to generate the structured query, then use this rule-based parser as a validation layer.

## Conclusion

This project successfully builds a CLI tool that converts natural language GitHub repository search requests into structured API parameters and executes them against the GitHub API.

The baseline tool can extract topics, programming languages, star constraints, dates, licenses, exclusions, sorting preferences, and result limits. The break-it phase identified important failure cases such as typos, vague queries, conflicting constraints, multilingual input, and API limit issues. The hardening phase improved the parser through normalization, typo correction, basic Chinese keyword translation, validation, and warning messages.

However, some remaining cases are fundamentally difficult because they require subjective judgment, semantic similarity, full multilingual understanding, or information that is not directly available from GitHub’s structured API fields. These limitations show why natural-language-to-query systems are challenging and why evaluation and hardening are important parts of building reliable AI tools.
