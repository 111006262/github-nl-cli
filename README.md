# GitHub Natural Language Repository Search CLI

## Part 1: Build a Tool, Break It, and Harden It

### Project Overview

This project is a command-line interface (CLI) tool that converts a natural language query into a structured GitHub repository search request, executes the request using the GitHub API, and returns real repository results.

The goal of the project is to explore how natural language can be converted into structured API parameters. The tool also includes a "break-it" phase, where ambiguous, misspelled, conflicting, and multilingual inputs are tested. After identifying these failure cases, the parser is hardened with validation rules, typo fixes, basic Chinese keyword translation, and warnings for impossible or vague queries.

Example user input:

```bash
python cli.py "Find the top 5 Python repositories about machine learning"
```

Example structured GitHub API request generated by the tool:

```json
{
  "q": "machine learning language:Python",
  "sort": "stars",
  "order": "desc",
  "per_page": "5"
}
```

The tool then sends this request to GitHub and prints the top matching repositories in a JSON format.

---

### Domain Choice

The chosen domain is the GitHub public repository dataset.

I chose GitHub because its repository search has useful structured fields that can be extracted from natural language, such as:

- programming language
- topic/search terms
- number of stars
- creation date
- update date
- license
- number of returned results
- sorting method

This then makes GitHub a good target for a natural-language-to-structured-query tool.

---

### GitHub API Query Format

The structured query format used in this project is based on GitHub's official repository search API and GitHub search syntax.

The GitHub repository search endpoint is:

```text
https://api.github.com/search/repositories
```

This endpoint accepts query parameters such as:

```text
- q
- sort
- order
- per_page
```

More information on the format structure are listed in this link:

```text
https://docs.github.com/en/rest/search/search?apiVersion=2022-11-28#search-repositories
```

### File-by-File Explanation

#### `cli.py`

`cli.py` is the main command-line interface file.

It is responsible for:

1. Reading the user’s natural language query from the terminal.
2. Sending that query to `parser.py`.
3. Printing the structured GitHub API request.
4. Checking for warnings.
5. Sending the structured query to `github_client.py`.
6. Printing the final GitHub repository results.

Example command:

```bash
python cli.py "Find the top 5 Python repositories about machine learning"
```

The CLI first prints the original natural language input:

```text
Natural language input:
  Find the top 5 Python repositories about machine learning
```

Then it prints the structured GitHub API request:

```json
{
  "q": "machine learning language:Python",
  "sort": "stars",
  "order": "desc",
  "per_page": "5"
}
```

If the query encounters problems such as being too vague or having conflicting constraints, the CLI stops before calling the GitHub API.

For example:

```bash
python cli.py "Show me popular repos"
```

This query is too vague because it does not include a topic, language, or useful constraint. The tool therefore prints a warning and stops.

---

#### `parser.py`

`parser.py` is the core natural language parser.

Its job is to convert the user’s natural language query into a structured query object. The parser is rule-based, which means it does not use an LLM. Instead, it uses regular expressions, dictionaries, and manually written extraction rules.

The main data structure is:

```python
@dataclass
class ParsedGitHubQuery:
    original_query: str
    search_terms: str
    language: Optional[str] = None
    min_stars: Optional[int] = None
    max_stars: Optional[int] = None
    created_after: Optional[str] = None
    created_before: Optional[str] = None
    pushed_after: Optional[str] = None
    license_name: Optional[str] = None
    exclude_terms: List[str] = field(default_factory=list)
    sort: str = "stars"
    order: str = "desc"
    per_page: int = 10
    warnings: List[str] = field(default_factory=list)
```

This object stores all extracted categories from the natural language query.

For example, the query:

```text
Find the top 5 Python repositories about machine learning with more than 1000 stars
```

is parsed into:

```text
search_terms = "machine learning"
language = "Python"
min_stars = 1000
sort = "stars"
order = "desc"
per_page = 5
```

Then the parser converts this into GitHub search syntax:

```text
machine learning language:Python stars:>1000
```

With this, the parser separates the query into categories using different helper functions.

##### `normalize_text()`

This function prepares the query before extraction.

It performs:

- basic whitespace cleanup
- simple typo correction
- simple Chinese keyword translation

For example:

```text
Find pyton repos about machien lerning
```

becomes:

```text
Find python repositories about machine learning
```

The parser includes dictionaries such as `TYPO_FIXES` and `CHINESE_TRANSLATIONS` to support some failure cases.

##### `extract_limit()`

This function detects how many results the user wants.

Examples:

```text
top 5
first 10
show 3
find 8
```

If a number is found, it becomes the `per_page` value. If no number is found, the default is 10.

##### `extract_language()`

This function detects programming languages.

It checks the query against the `LANGUAGES` dictionary. For example:

```text
Python
JavaScript
TypeScript
Rust
Go
C++
```

If the user writes:

```text
Find Rust repositories about blockchain
```

the parser extracts:

```text
language = "Rust"
```

Note that special handling is included for `Go` because `go` can also be an ordinary English verb.

##### `extract_stars()`

This function detects star constraints.

Examples:

```text
more than 1000 stars
over 500 stars
at least 200 stars
less than 100 stars
under 50 stars
```

For example:

```text
Find Python repositories with more than 1000 stars
```

becomes:

```text
stars:>1000
```

##### `extract_dates()`

This function detects date-related constraints.

Examples:

```text
created after 2021
created before 2020
updated this year
last year
```

For example:

```text
Find Rust blockchain projects created after 2021
```

becomes:

```text
created:>2021-12-31
```

##### `extract_license()`

This function detects license constraints.

Examples:

```text
MIT license
Apache license
GPL license
BSD license
```

For example:

```text
Find repositories about deep learning with MIT license
```

becomes:

```text
license:mit
```

##### `extract_exclusions()`

This function handles negative constraints.

Examples:

```text
not machine learning
without tensorflow
```

For example:

```text
Find AI repositories but not machine learning
```

becomes:

```text
ai -machine-learning
```

##### `infer_sort()`

This function decides how the GitHub results should be sorted.

Examples:

```text
newest
recently updated
most forks
forked
```

If no special sorting phrase is found, the default sorting method is by stars in descending order.

##### `clean_search_terms()`

After all structured fields are extracted, this function removes command words and filter words from the query. The remaining words become the main search topic.

For example:

```text
Find the top 5 Python repositories about machine learning with more than 1000 stars
```

The parser removes:

```text
find
top 5
python
repositories
about
with more than 1000 stars
```

The remaining topic is:

```text
machine learning
```

##### `validate_query()`

This function is the hardening layer for failure test cases.

It checks for:

- vague queries
- conflicting star constraints
- conflicting date constraints
- invalid result limits
- GitHub per-page limit issues

For example:

```text
Find repositories with more than 5000 stars and less than 100 stars about AI
```

This is impossible because a repository cannot have both more than 5000 stars and fewer than 100 stars. The parser detects this and adds a warning instead of blindly sending a bad request.

---

#### `github_client.py`

`github_client.py` is responsible for communicating with the GitHub API.

It receives structured API parameters from `parser.py` and sends them to:

```text
https://api.github.com/search/repositories
```

The main function is:

```python
search_repositories(params)
```

This function:

1. Creates request headers.
2. Optionally uses a GitHub token from the environment.
3. Sends a GET request to GitHub.
4. Raises an error if GitHub returns a non-200 response.
5. Returns the JSON response.

The file also includes:

```python
simplify_results(api_response)
```

GitHub returns a large JSON object, so this function keeps only the fields needed for display:

- repository full name
- description
- stars
- forks
- programming language
- repository URL
- last updated time

This makes the CLI output easier to read.

### Harden-and-Fix Phase

The parser was improved to handle the most important failure cases, mostly from typos to simple chinese to english translations, as well as ambiguous inputs.

#### 1. Typo Handling

The parser includes a small typo correction dictionary.

Example:

```text
pyton → python
machien → machine
lerning → learning
```

Input:

```text
Find pyton repos about machien lerning
```

Corrected interpretation:

```text
machine learning language:Python
```

This makes the tool more robust against common spelling mistakes.

---

#### 2. Basic Chinese Keyword Support

The parser includes a small Chinese-to-English keyword dictionary.

Examples:

```text
機器學習 → machine learning
人工智慧 → artificial intelligence
深度學習 → deep learning
電腦視覺 → computer vision
專案 → repository
```

Input:

```text
找出關於機器學習的 Python 專案
```

Parsed meaning:

```text
machine learning language:Python
```

This does not provide full multilingual understanding, but it supports common Chinese examples relevant to the GitHub repository search domain.

---

#### 3. Ambiguity Detection

The tool detects vague queries.

Example:

```text
Show me popular repos
```

This query does not include a topic, language, star condition, date condition, or any useful search constraint. Instead of sending a weak search request to GitHub, the parser adds a warning:

```text
The query is too vague. Please include a topic, language, or constraint.
```

The CLI then stops and asks for clarification.

---

#### 4. Conflict Detection

The parser detects impossible star constraints.

Example:

```text
Find repositories with more than 5000 stars and less than 100 stars about AI
```

This creates:

```text
min_stars = 5000
max_stars = 100
```

Since the minimum is greater than the maximum, the parser adds a warning:

```text
Conflicting star constraints: minimum stars is greater than or equal to maximum stars.
```

The CLI stops before calling GitHub.

---

#### 5. Result Limit Protection

GitHub search requests should not ask for unlimited results in a single call. The parser protects against this by capping the result count.

Example:

```text
Find top 200 Python repositories about data science
```

The parser caps the result limit to 100.

---

#### 6. Sorting Improvements

The parser supports different ranking intentions.

Examples:

```text
newest TypeScript chatbot repositories
```

Sorts by:

```text
created desc
```

```text
recently updated computer vision repositories
```

Sorts by:

```text
updated desc
```

```text
most forked Python repositories
```

Sorts by:

```text
forks desc
```

If the query does not specify a special sorting method, the default is:

```text
stars desc
```

This means the parser treats popularity as most-starred repositories.

### Remaining Failure Cases and Why They Are Fundamentally Hard

Even after hardening the parser, some failure cases remain difficult to solve completely, especially without an LLM.

#### 1. Subjective Queries

Example:

```text
Find the best repositories for beginners learning AI.
```

This is difficult because words like `best` and `beginner-friendly` are subjective. The GitHub API can sort search queries by measurable fields like stars, forks, creation date, or update date, but it cannot directly determine whether a repository is truly beginner-friendly.

A repository with many stars is not always the best for beginners. It may be popular but poorly documented, outdated, or too advanced. Solving this properly would require deeper analysis of README files, tutorials, documentation quality, issue activity, and possibly user feedback.

Therefore, this is fundamentally hard because the user’s intent depends on qualitative judgment, while the GitHub API mostly supports quantitative filters.

---

#### 2. Complex Semantic Similarity

Example:

```text
Find repositories similar to TensorFlow but easier to use.
```

This is hard because GitHub search does not directly support semantic similarity. The parser can search for the word `TensorFlow`, but it cannot truly understand what makes another repository similar to TensorFlow.

The phrase `easier to use` is also difficult because usability is not a simple structured field in the GitHub API. Solving this would require repository embeddings, README analysis, documentation analysis, or comparison against known machine learning frameworks.

This is fundamentally hard because the required information is not directly available as a simple API filter.

---

#### 3. Full Multilingual Understanding

Example:

```text
Pythonで機械学習の人気リポジトリを探して
```

The current parser only supports a small dictionary of Chinese keyword translations. It does not support full Japanese, Korean, Indonesian, French, or mixed-language natural language queries.

Full multilingual support is difficult because users may phrase the same request in many different ways. A dictionary-based approach cannot cover all grammar patterns, synonyms, and languages.

Solving this properly would require a translation model, multilingual LLM, or a much larger multilingual parser.

---

#### 4. Ambiguous Words

Example:

```text
Find Go projects about web servers.
```

In this case, `Go` probably means the Go programming language. However, in other sentences, `go` could simply be an English verb.

Example:

```text
I want to go find repositories about AI.
```

A simple rule-based parser may incorrectly interpret `go` as the programming language. The current parser includes special handling for Go, but ambiguous words are still generally difficult.

This is fundamentally hard because correct interpretation depends on sentence context, not just keyword matching.

---

#### 5. Multi-Intent Queries

Example:

```text
Find Python machine learning repositories that are actively maintained, beginner-friendly, and not too large.
```

This query contains multiple abstract requirements:

- actively maintained
- beginner-friendly
- not too large

The parser can approximate `actively maintained` using `recently updated`, but it cannot fully judge maintenance quality. It also cannot determine beginner-friendliness or project size without deeper analysis of repository contents.

This is hard because the query asks for judgments that are not directly represented in the GitHub search API.

---

#### 6. Hidden User Assumptions

Example:

```text
Find good repositories for my final project.
```

This is difficult because the tool does not know what the user’s final project is about. The query is meaningful to the user, but not specific enough for the parser.

There is usually a follow-up question to this, such as:

```text
What is your final project topic?
```

The current CLI only warns when the query is too vague. It does not have a full conversation system for asking follow-up questions.

This is fundamentally hard because the missing information exists outside the query itself.

---

### Technical Design Trade-Offs

This project uses a rule-based parser instead of an LLM for Part 1.

The advantage of this approach is that it is:

- explainable
- easy to debug
- deterministic
- inexpensive
- simple to run locally

The disadvantage is that it is less flexible than an LLM. It can only handle the patterns that were manually implemented.

For example, the parser can understand:

```text
more than 1000 stars
```

but may fail on unusual phrasings such as:

```text
repos that have gained at least a thousand stars
```

A future version could use an LLM to generate the structured query, then use this rule-based parser as a validation layer.

### Conclusion

This project successfully builds a CLI tool that converts natural language GitHub repository search requests into structured API parameters and executes them against the GitHub API.

The baseline tool can extract topics, programming languages, star constraints, dates, licenses, exclusions, sorting preferences, and result limits. The break-it phase identified important failure cases such as typos, vague queries, conflicting constraints, multilingual input, and API limit issues. The hardening phase improved the parser through normalization, typo correction, basic Chinese keyword translation, validation, and warning messages.

However, some remaining cases are fundamentally difficult because they require subjective judgment, semantic similarity, full multilingual understanding, or information that is not directly available from GitHub’s structured API fields. These limitations show why natural-language-to-query systems are challenging and why evaluation and hardening are important parts of building reliable AI tools.


## Part 2: The Multi-Model Eval Challenge

### Objective

Part 2 evaluates how well different language models can convert natural language GitHub repository search requests into the same structured query format used in Part 1. Instead of only checking whether the CLI works, this part tests whether multiple LLMs can consistently produce correct GitHub API parameters.

The required structured output format is:

```json
{
  "q": "string",
  "sort": "stars | forks | updated | created",
  "order": "asc | desc",
  "per_page": "string number"
}
```

The four fields are important because they directly control the GitHub repository search request. This includes the following:

- `q`: the actual GitHub search query, including keywords and filters such as `language:Python`, `stars:>1000`, `created:>2021-12-31`, or `pushed:>2026-01-01`
- `sort`: the ranking method, such as stars, forks, updated, or created
- `order`: ascending or descending order
- `per_page`: the number of results requested, capped at 100

#### `generate_eval_data.py`

This file programmatically creates the evaluation dataset. It writes 30 test cases into `eval_cases.json`. Each test case includes a natural language input and a manually written ground truth structured query.

Example:

```json
{
  "id": "case_01",
  "input": "Find the top 5 Python repositories about machine learning",
  "ground_truth": {
    "q": "machine learning language:Python",
    "sort": "stars",
    "order": "desc",
    "per_page": "5"
  }
}
```

#### `eval_cases.json`

This file stores the 30 evaluation cases. The cases are intentionally diverse and include:

- simple GitHub search queries
- programming language filters
- star constraints
- date constraints
- license constraints
- result count constraints
- typo-heavy inputs
- Chinese input
- negative constraints
- ambiguous words such as `Go`
- sorting instructions such as `newest`, `updated this year`, and `most forked`

#### `model_clients.py`

This file connects the evaluation pipeline to the three selected models:

- Gemini 2.5 Flash-Lite 
- Qwen2.5 7B through Ollama
- Mistral 7B through Ollama

It also contains the shared system prompt, JSON schema, model call functions, and the targeted validation-retry logic used for Mistral.

#### `scorer.py`

This file compares model outputs against the ground truth. A prediction is correct only when all four fields match after normalization:

- `q`
- `sort`
- `order`
- `per_page`

The scorer intentionally normalizes formatting differences. For example:

```text
React = react
language:rust = language:Rust
language=R = language:R
machine-learning = machine learning
web-scraping = web scraping
```

However, the scorer does not ignore meaningful differences. For example, these are still considered wrong:

```text
updated:>2026-01-01 instead of pushed:>2026-01-01
created:>2022 instead of created:>2022-12-31
sort:created instead of sort:stars
per_page:200 instead of per_page:100
```

This then made the scoring more fair without allowing incorrect API behavior to pass, but also considers slight mismatches including capitalization and dashes in the normal search queries.

#### `eval_models.py`

This is the main evaluation runner as it loads the evaluation cases, sends each natural language input to a selected model, scores the response, and saves the full results as JSON.

Example commands:

```bash
python eval_models.py --model gemini
python eval_models.py --model qwen
python eval_models.py --model mistral
python eval_models.py --model all
```

### Evaluation Dataset and Ground Truth Design

The evaluation dataset contains 30 manually labeled examples. The ground truth was written by deciding the correct GitHub API interpretation for each natural language input, testing different failure cases we might encounter, as similar to Part 1.

Examples of ground truth design decisions:

#### Date filters

```text
Find Rust blockchain projects created after 2021
```

Ground truth:

```json
{
  "q": "blockchain language:Rust created:>2021-12-31",
  "sort": "stars",
  "order": "desc",
  "per_page": "10"
}
```

The phrase `created after 2021` means the repository should be created after the end of 2021, so the query uses:

```text
created:>2021-12-31
```

#### Updated repositories

```text
Find repositories about computer vision updated this year
```

Ground truth:

```json
{
  "q": "computer vision pushed:>2026-01-01",
  "sort": "updated",
  "order": "desc",
  "per_page": "10"
}
```

GitHub repository search uses `pushed:` for update activity, so the correct filter is:

```text
pushed:>2026-01-01
```

not:

```text
updated:>2026-01-01
```

#### Result limit cap

```text
Find top 200 Python repositories about data science
```

Ground truth:

```json
{
  "q": "data science language:Python",
  "sort": "stars",
  "order": "desc",
  "per_page": "100"
}
```

Even though the user asks for 200 results, the API request is capped at 100 results per page.

#### Negative constraints

```text
Find AI repositories but not machine learning
```

Ground truth:

```json
{
  "q": "ai -machine-learning",
  "sort": "stars",
  "order": "desc",
  "per_page": "10"
}
```

The negative phrase `not machine learning` is represented using a dash:

```text
-machine-learning
```

### Scoring Method

The evaluator compares each model prediction to the ground truth field by field.

```python
field_scores = {
    "q": q_tokens(pred["q"]) == q_tokens(truth["q"]),
    "sort": pred["sort"] == truth["sort"],
    "order": pred["order"] == truth["order"],
    "per_page": pred["per_page"] == truth["per_page"],
}
```

A case is counted as correct only if all four fields are correct.

The `q` field is normalized before comparison. This was necessary because GitHub search is not usually affected by simple capitalization or token order. For example, these should be treated as equivalent:

```text
machine learning language:Python
language:Python machine learning
```

The scorer also treats ordinary hyphenated topic phrases as equivalent to space-separated phrases:

```text
machine-learning = machine learning
web-scraping = web scraping
natural-language-processing = natural language processing
```

However, negative terms are preserved:

```text
-machine-learning
```

and is not converted into:

```text
machine learning
```

This distinction matters because a dash at the start of a term changes the meaning from inclusion to exclusion.

### Performance Comparison and Initial Error Patterns

#### Gemini

Gemini performed the best in terms of speed and consistency. It followed the JSON schema well and usually produced the correct GitHub query syntax. It initially reached a score of 27/30 (90%) in the first try, and after making changes to the pipeline and syntax adjusted for the local models, it reached a score of 28/30, or 93.33% accuracy.

The main problems Gemini still had were small date-format or range-format differences. For example, one remaining issue was using GitHub's range syntax for a year range rather than the exact ground truth format:

```text
created:2025-01-01..2025-12-31
```

instead of:

```text
created:>2025-01-01 created:<2025-12-31
```

This shows that Gemini understood the meaning, but it did not always match the exact expected structured query style.

I also encountered Gemini API quota issues during testing. Even after setting a spending cap, Gemini could still hit requests-per-minute or requests-per-day limits. This showed that billing limits and API rate limits are separate. To avoid unnecessary charges, I later disabled or removed the Gemini API key after completing the evaluation.

#### Qwen2.5 7B

Qwen initially achieved 23/30, or 76.67%, before the scoring changes. Most of its mistakes were not complete misunderstandings. Instead, they were small formatting or default-value issues, such as:

- using `language:rust` instead of `language:Rust`
- using `language=R` instead of `language:R`
- capitalizing ordinary search terms like `React` or `Android`
- adding an extra language filter such as `language:Python`
- setting `per_page` incorrectly in some cases

At first, I experimented with a `repair_prediction()` function that modified the model prediction before scoring. However, this made the output look too close to the ground truth and was not a fair evaluation of the model. Its function was to modify the prediction based on what I have created in Part 1, but I think this defeats the purpose of using LLM models to predict the queries. With this, I removed that approach and instead changed the scorer to normalize harmless differences only.

After the scorer normalization, Qwen reached 27/30, or 90.00% accuracy.

#### Mistral 7B

Mistral was the slowest model and initially performed the worst. Its first result was 12/30, or 40.00%. The biggest issue was that Mistral often hyphenated ordinary multi-word topic phrases:

```text
machine-learning
web-scraping
data-science
natural-language-processing
```

instead of:

```text
machine learning
web scraping
data science
natural language processing
```

Since these are usually equivalent search terms, I updated the scorer to treat normal hyphenated topic phrases as equivalent to space-separated phrases. However, the scorer still preserves negative terms such as `-tensorflow` and `-machine-learning` because those are meaningful exclusions.

After scorer normalization, Mistral improved to 20/30, or 66.67%. To make the model more accurate, I then tried adding more rules to the system prompt. After adding important date/sort rules to the prompt, it improved to 22/30, or 73.33%. It improved by a few percent, however, I did notice that the model loaded slower. With this, I took off the extra prompt and decided to add a targeted validation-retry step for this specific model. This step checks the model output against known rules using only the input query and the model prediction (not the ground truth). If the model violates a rule, it asks Mistral to correct its own JSON. The result of this still retained a 22/30 score or 73.33% accuracy. With this, I decided to combine both the additional system prompt with the targeted validation-retry step.

After this retry step, Mistral reached 29/30, or 96.67% accuracy.

### Iterations and Code Changes Made During the Process

#### 1. Added structured JSON schema

The model prompt tells each model to return only JSON with four required keys:

```json
{
  "q": "string",
  "sort": "string",
  "order": "string",
  "per_page": "string"
}
```

This reduced invalid outputs and made the responses easier to score automatically.

#### 2. Added scorer normalization for capitalization

Early results showed that the scorer was too strict. It marked outputs wrong even when the meaning was the same.

Examples:

```text
React vs react
Android vs android
language:rust vs language:Rust
language=R vs language:R
```

I updated `scorer.py` so that harmless capitalization and syntax differences are normalized before comparison.

#### 3. Added scorer normalization for normal hyphenated topic phrases

Mistral often produced:

```text
machine-learning
web-scraping
```

The ground truth used:

```text
machine learning
web scraping
```

I updated the scorer so that ordinary hyphenated topic phrases are treated as equivalent to space-separated terms. However, negative terms are still preserved.


#### 4. Added important date and sorting rules to the prompt

Mistral often confused date filters with sorting. For example, it sometimes changed `sort` to `created` simply because the query included a `created after` filter.

To address this, I added rules such as:

```text
A date filter does not automatically change the sort field.
Only use sort "created" when the user says "newest" or "recently created".
For "updated this year", use pushed:>2026-01-01, not updated:>2026-01-01.
```

This improved Mistral, but not enough by itself.

#### 5. Added targeted validation-retry for Mistral

Because making the main prompt longer made the model slower and still did not fix all mistakes, I added a targeted retry system for Mistral.

The retry system works like this:

```text
Mistral gives first JSON output
- validator checks for known rule violations
- if there is no issue, keep the output
- if there is an issue, send feedback to Mistral
- Mistral returns corrected JSON
- scorer evaluates the final JSON
```

The validator checks for issues such as:

- using `updated:>` instead of `pushed:>`
- using `created:>2022` instead of `created:>2022-12-31`
- adding unnecessary created-date filters for `newest`
- rewriting `AI` as `artificial intelligence`
- adding extra frameworks or OR clauses that were not requested
- missing `language:Java` when the user asks for Java repositories

This is more fair than directly repairing the prediction because the final corrected output still comes from the model. The validator should not and does not read the ground truth file.

### Learnings

#### 1. The evaluation design is subjective but had to be well-thought-of

At first, it seemed like the main task was just to call three models and check their outputs to compare them against each other. However, the harder part was designing a fair evaluation method. As there is no specific rules to how we should personally score these outputs, and build the ground truths, the scorer had to be strict enough to catch real mistakes, but also flexible enough to avoid penalizing formatting differences as each LLM has their own way of understanding prompts.

#### 2. The ground truth needs clear interpretation rules

Some natural language inputs can be interpreted in more than one way. For example:

```text
created after 2021
```

could be interpreted loosely as:

```text
created:>2021
```

or more precisely as:

```text
created:>2021-12-31
```

For this project, I decided to define one consistent interpretation and use it across all models. After deciding to define one interpretation, I think that the format we created should be specified strictly in the prompt to make answers more uniformed.

#### 3. Small structured-output differences can change API behavior

In normal text generation, small wording differences may not matter much. However, in structured query generation, small differences can completely change the API request.

For example:

```text
pushed:>2026-01-01
```

and:

```text
updated:>2026-01-01
```

both seems similar in meaning, but only `pushed:` is the correct GitHub search qualifier for recently updated repositories.

Similarly:

```text
sort:created
```

and:

```text
sort:stars
```

these return very different result rankings in GitHub.

#### 4. A longer prompt is not always better

Adding more rules helped somewhat, especially for Mistral's date and sort mistakes. However, longer prompts made the local models slower and did not always solve the problem. The targeted validation-retry step was more efficient in this case because it is only triggered when the first output violated known rules.

#### 5. Local open-weight models can work, but it needs more support

Qwen and Mistral were able to pass the 85% threshold, but they needed more evaluation pipeline support than Gemini. Qwen mainly needed fair scorer normalization, while Mistral needed both scorer normalization and a validation-retry step.

#### 6. The evaluation pipeline itself is part of the system as a whole

This project showed that model performance is not only about improving the model. The prompt, schema, scorer, normalization, validation, and retry logic all affect final accuracy. Building a rigorous evaluation pipeline means documenting all of these choices clearly, and makes it more uniformed for any model that would be used.
